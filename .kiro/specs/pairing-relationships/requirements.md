# Requirements Document: Pairing & Relationships System

## Introduction

The Pairing & Relationships system enables parents and caregivers to establish secure connections within the PillSathi medication reminder app. Parents generate time-limited invite codes that caregivers can use to establish relationships. The system supports many-to-many relationships, allowing parents to have multiple caregivers and caregivers to support multiple parents. This feature builds on Phase 1 (Authentication) where users have already selected their role (Parent or Caregiver).

## Glossary

- **Parent**: A user with the "parent" role who takes medications and can generate invite codes
- **Caregiver**: A user with the "caregiver" role who helps monitor medication adherence for parents
- **Invite_Code**: A time-limited alphanumeric code generated by a parent to enable caregiver pairing
- **Relationship**: A bidirectional connection between a parent and a caregiver
- **Pairing_System**: The system component that manages invite code generation, validation, and relationship creation
- **Relationship_Manager**: The system component that manages viewing and removing relationships
- **TTL**: Time-to-live, the duration for which an invite code remains valid
- **Firestore**: The Firebase database service used for data persistence
- **Cloud_Function**: Server-side Firebase function for secure operations

## Requirements

### Requirement 1: Invite Code Generation

**User Story:** As a parent, I want to generate an invite code, so that I can share it with caregivers to establish a connection.

#### Acceptance Criteria

1. WHEN a parent requests invite code generation, THE Pairing_System SHALL create a unique alphanumeric code
2. WHEN an invite code is created, THE Pairing_System SHALL store it in Firestore with the parent's user ID, creation timestamp, and expiration timestamp
3. WHEN an invite code is created, THE Pairing_System SHALL set the TTL to 24 hours from creation time
4. WHEN an invite code is generated, THE Pairing_System SHALL return the code to the parent within 3 seconds
5. WHERE a parent already has an active unexpired invite code, THE Pairing_System SHALL return the existing code instead of creating a new one

### Requirement 2: Invite Code Sharing

**User Story:** As a parent, I want to share my invite code with caregivers, so that they can link to my account.

#### Acceptance Criteria

1. WHEN a parent views their invite code, THE Pairing_System SHALL display the code in a readable format
2. WHEN a parent requests to share the code, THE Pairing_System SHALL provide native sharing options for the device
3. WHEN displaying an invite code, THE Pairing_System SHALL show the remaining time until expiration
4. IF an invite code has expired, THEN THE Pairing_System SHALL display an expiration message and offer to generate a new code

### Requirement 3: Invite Code Redemption

**User Story:** As a caregiver, I want to enter an invite code to link with a parent, so that I can help monitor their medication adherence.

#### Acceptance Criteria

1. WHEN a caregiver submits an invite code, THE Cloud_Function SHALL validate the code format before processing
2. WHEN a valid invite code is submitted, THE Cloud_Function SHALL verify the code exists in Firestore
3. IF the invite code has expired, THEN THE Cloud_Function SHALL reject the redemption and return an expiration error
4. IF the invite code does not exist, THEN THE Cloud_Function SHALL reject the redemption and return an invalid code error
5. WHEN a valid unexpired invite code is redeemed, THE Cloud_Function SHALL create a relationship record linking the caregiver to the parent
6. WHEN a relationship is created, THE Cloud_Function SHALL store both user IDs and a creation timestamp in Firestore
7. IF a relationship already exists between the caregiver and parent, THEN THE Cloud_Function SHALL return a success response without creating a duplicate

### Requirement 4: Relationship Viewing for Caregivers

**User Story:** As a caregiver, I want to see all my linked parents, so that I can access their medication information.

#### Acceptance Criteria

1. WHEN a caregiver views their relationships, THE Relationship_Manager SHALL query Firestore for all relationships where the caregiver is a participant
2. WHEN displaying relationships, THE Relationship_Manager SHALL show the parent's name and profile information
3. WHEN relationship data changes, THE Relationship_Manager SHALL update the display in real-time using Firestore listeners
4. WHEN a caregiver has no relationships, THE Relationship_Manager SHALL display an empty state with instructions to enter an invite code

### Requirement 5: Relationship Viewing for Parents

**User Story:** As a parent, I want to see all my linked caregivers, so that I know who has access to my medication information.

#### Acceptance Criteria

1. WHEN a parent views their relationships, THE Relationship_Manager SHALL query Firestore for all relationships where the parent is a participant
2. WHEN displaying relationships, THE Relationship_Manager SHALL show the caregiver's name and profile information
3. WHEN relationship data changes, THE Relationship_Manager SHALL update the display in real-time using Firestore listeners
4. WHEN a parent has no relationships, THE Relationship_Manager SHALL display an empty state with instructions to generate an invite code

### Requirement 6: Relationship Removal

**User Story:** As either a parent or caregiver, I want to remove a relationship, so that I can revoke access when needed.

#### Acceptance Criteria

1. WHEN a user requests to remove a relationship, THE Relationship_Manager SHALL display a confirmation dialog
2. WHEN a user confirms relationship removal, THE Cloud_Function SHALL delete the relationship record from Firestore
3. WHEN a relationship is removed, THE Relationship_Manager SHALL update both users' relationship lists in real-time
4. WHEN a relationship removal fails, THE Relationship_Manager SHALL display an error message and maintain the current state

### Requirement 7: Data Security and Access Control

**User Story:** As a system administrator, I want to ensure relationship data is secure, so that users' privacy is protected.

#### Acceptance Criteria

1. THE Firestore_Security_Rules SHALL prevent users from reading invite codes they did not create
2. THE Firestore_Security_Rules SHALL prevent users from reading relationships they are not part of
3. THE Firestore_Security_Rules SHALL prevent users from creating relationships directly without using the Cloud_Function
4. THE Firestore_Security_Rules SHALL allow users to delete only relationships they are part of
5. THE Cloud_Function SHALL validate that the requesting user is authenticated before processing any relationship operations

### Requirement 8: Invite Code Expiration and Cleanup

**User Story:** As a system administrator, I want expired invite codes to be cleaned up, so that the database remains efficient.

#### Acceptance Criteria

1. WHEN an invite code expires, THE Pairing_System SHALL mark it as invalid for redemption
2. THE Cloud_Function SHALL automatically delete invite codes that are older than 48 hours
3. WHEN checking invite code validity, THE Pairing_System SHALL compare the current timestamp against the expiration timestamp

### Requirement 9: Error Handling and User Feedback

**User Story:** As a user, I want clear feedback when operations fail, so that I understand what went wrong and how to fix it.

#### Acceptance Criteria

1. WHEN an invite code redemption fails, THE Pairing_System SHALL display a specific error message indicating the failure reason
2. WHEN a network error occurs, THE Pairing_System SHALL display a retry option
3. WHEN a relationship operation succeeds, THE Pairing_System SHALL display a success confirmation
4. WHEN loading relationship data, THE Pairing_System SHALL display a loading indicator
